Engineering

핵심 기술 결정

프로젝트의 주요 설계 결정과 그 이유입니다.

1

Supabase PostgreSQL (vs Firebase/MongoDB)

SQL의 유연성, Row Level Security로 사용자별 데이터 격리, 트리거 기반 자동화가 가능한 Supabase를 선택했습니다. 인증/스토리지/실시간 구독을 하나의 서비스에서 제공하여 인프라 복잡도를 줄였습니다.

2

3단계 식품 검색 파이프라인

DB 캐시 → 정부 API (식약처/Open Food Facts) → AI 추론(GPT-4o-mini)의 단계적 조회로, 응답 속도와 정확도를 모두 확보하면서 API 비용을 최소화했습니다.

3

GPT-4o-mini (vs GPT-4 Turbo)

이미지 분석과 증상 판단에 GPT-4o-mini를 사용하여 비용 효율을 극대화했습니다. 일일 사용량 제한(Rate Limiting)과 결합하여 서비스 운영 비용을 관리합니다.

4

SSE 기반 실시간 관리자 대시보드 (vs WebSocket)

관리자 접속자 모니터링에 Server-Sent Events를 사용했습니다. 서버 → 클라이언트 단방향 통신만 필요하므로 WebSocket보다 가볍고, 서버리스 환경(Vercel)과의 호환성도 좋습니다.

5

Capacitor (vs React Native/Flutter)

기존 웹 코드베이스를 그대로 재사용하면서 네이티브 앱으로 배포할 수 있는 Capacitor를 선택했습니다. 별도의 네이티브 코드 없이 카메라, 햅틱, 상태바 API를 활용합니다.

6

한국 특화 최적화

Naver Maps (한국에서 Google Maps보다 높은 커버리지), Kakao OAuth (한국 사용자 보급률 1위), NEIS 학교급식 API, 초성 검색 등 한국 시장에 최적화된 서비스를 연동했습니다.
